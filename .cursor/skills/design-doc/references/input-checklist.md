# Design Document Input Checklist

Inspect inputs before drafting the Design Document.

## Read order

1. Repository rules and project instructions
2. `docs/inputs/project-brief.md`
3. `docs/inputs/brand-guidelines.md`
4. `docs/inputs/content-inventory.md`
5. `docs/inputs/audience-and-offer.md`
6. `docs/inputs/business-requirements.md`
7. `docs/inputs/technical-constraints.md`
8. `docs/inputs/analytics-and-conversion.md`
9. `docs/inputs/legal-and-compliance.md`
10. `docs/inputs/research.md`
11. Existing designs, decisions, specs, source code, and deployment configuration

## Input classification

### Required

- Project brief
- Brand guidelines or explicit visual direction
- Content inventory
- Business requirements
- Technical constraints or repository inspection

### Recommended

- Audience and offer
- Analytics and conversion
- Legal and compliance
- Research

### Optional

- Existing wireframes
- Competitor references
- User interviews
- Campaign reports
- Approved image library

## Input status table

Create this table before drafting:

| Input | Found | Status | Conflicts | Action |
|---|---|---|---|---|

Use these status values:

- `usable`
- `partial`
- `missing`
- `conflicting`
- `requires-verification`

## Missing information policy

- Mark missing facts as `[OPEN]`.
- Mark provisional interpretations as `[ASSUMPTION]`.
- Mark non-binding suggestions as `[RECOMMENDATION]`.
- Never silently convert a recommendation into an approved requirement.
- Continue with non-critical gaps; block approval when a gap affects legality, professional scope, primary conversion, or technical feasibility.
